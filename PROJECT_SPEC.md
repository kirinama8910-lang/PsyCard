
# PROJECT_SPEC.md
## Проект: psy-client-cards

---

# 1. Назначение проекта

`psy-client-cards` — это локальное приложение для психолога для ведения карт клиентов в формате `.docx`.

Приложение должно:

- работать только локально на компьютере психолога
- принимать текстовые и голосовые заметки
- распознавать клиента по первой строке заметки
- искать существующую карту клиента
- если карты нет — создавать новую по шаблону
- если карта есть — вносить изменения только в нужные секции
- использовать LLM для структурирования заметок и плана работы
- показывать preview перед записью
- делать backup перед записью
- вести логи изменений

Проект предназначен только для работы с клиентскими картами и не должен хранить данные в облачной базе.

---

# 2. Основные принципы проекта

## 2.1 Локальность

Приложение запускается только локально через Streamlit.

Адрес запуска:

`http://127.0.0.1:8501`

Никакого внешнего доступа и публичного деплоя на этом этапе.

---

## 2.2 Конфиденциальность и обработка данных

Все карты клиентов, шаблоны, заметки, бэкапы и логи хранятся локально внутри папки проекта.

### Обработка данных через LLM API

Для структурирования заметок текст клиентских записей отправляется в облачный API (Anthropic Claude). Важно учитывать:

* Anthropic API не использует данные из API-запросов для обучения моделей
* Тем не менее, данные передаются через интернет — это нужно иметь в виду при работе с особо чувствительной информацией
* В будущем возможно добавление локальной модели (Ollama) как альтернативы для полной автономности

Психолог должен быть осведомлён о том, что заметки обрабатываются через внешний API.

---

## 2.3 Безопасность записи

Карта клиента не должна переписываться целиком без необходимости.

Правильная логика:

1. приложение получает заметку
2. определяет клиента
3. определяет тип задачи (пользователь выбирает вручную)
4. ищет карту
5. если карты нет — создаёт новую по шаблону
6. LLM возвращает структурированный JSON
7. приложение показывает preview изменений
8. создаёт backup
9. вносит изменения только в нужные поля/секции
10. сохраняет результат безопасно

---

## 2.4 Relative paths only

Нельзя использовать абсолютные пути вида:

`C:\Users\...`

Нужно использовать только относительные пути через `pathlib.Path`, чтобы проект можно было передать целиком другому человеку.

### Отличия от LLM-архитектуры с двумя провайдерами

Вместо раздельных `claude_provider.py` и `openai_provider.py` используется единый `llm_provider.py` с одним провайдером (Claude). Это упрощает:

* обработку ошибок (один формат)

Это избавляет от необходимости синхронизировать отдельный файл метаданных с реальным состоянием файловой системы.

# 4. Хранение клиентов и правила именования

## 4.1 Идентификатор клиента

Клиент идентифицируется строкой формата:

`Имя 1234`

где:

* `Имя` — имя клиента
* `1234` — последние 4 цифры телефона

Примеры:

* `Анна 2332`
* `Наталья 3445`

---

## 4.2 Правило входных заметок

Каждая текстовая заметка или текст после расшифровки голосового файла должна начинаться с первой строки:

`Имя 1234`

Пример:

```text
Анна 2332
• Пол и возраст: женщина, 36 лет
• Жалобы: ...
• Запрос (словами клиента): ...
```

---

## 4.3 Правило валидации клиента

На первом этапе валидируется строгий формат:

* имя
* один пробел
* 4 цифры подряд

Пример регулярного выражения:

```regex
^([А-Яа-яЁёA-Za-z\-]+)\s(\d{4})$
```

Если первая строка не проходит валидацию, приложение не должно записывать данные в карту.

---

## 4.4 Правило хранения карты клиента

Каждый клиент хранится в отдельной папке.

Пример:

```text
data/clients/anna_2332/
└─ Анна 2332.docx
```

Имя папки строится как `имя_цифры` в нижнем регистре (транслитерация кириллицы в латиницу).

# 5. Шаблон карты клиента

В проекте должен использоваться шаблон карты:

`data/templates/card_template.docx`

### Требования к формату шаблона

Чтобы приложение могло надёжно находить и обновлять секции:

* Секции обозначаются заголовками уровня Heading 1 или Heading 2
* Поля внутри секций — параграфы с жирным ключом и обычным значением (например: **Жалобы:** текст)
* Не использовать сложное форматирование: вложенные таблицы, текстовые боксы, объединённые ячейки
* Допустимы простые таблицы и нумерованные/маркированные списки

### Версионирование формата

Шаблон содержит скрытый параграф или свойство документа `format_version: 1`. При изменении структуры шаблона версия увеличивается. Приложение предупреждает, если карта клиента создана по старой версии шаблона.

### Создание карты нового клиента

Если карта клиента не найдена, приложение должно:

1. взять шаблон
2. создать папку клиента
3. скопировать шаблон
4. назвать новую карту по правилу: `Имя 1234.docx`
5. заполнить первичные поля
6. сохранить новую карту

# 7. Типы входящих заметок и выбор task_type

## 7.1 Как определяется тип задачи

Пользователь выбирает тип задачи вручную через интерфейс (радиокнопка или выпадающий список в Streamlit). Доступные варианты:

* **Первичное заполнение / обновление карты** (`intake_create_or_update`)
* **Заметка после встречи** (`session_update`)
* **Методическое обновление плана** (`client_plan_update`)

Автоматическое определение типа может быть добавлено позже как удобство, но не является обязательным для первого релиза.

---

## 7.2 Первичное заполнение карты нового клиента

Пример:

```text
Анна 2332
• Пол и возраст: женщина, 36 лет
• Жалобы: Амбивалентные чувства, не понимаю себя, нет любви к близким, теряю себя, когда сыпется критика
• Запрос (словами клиента): Разобраться в себе
• Выявленный симптом: ожидание помощи от окружающих
• Дополнения из жизни: незавершенный развод с мужем...
```

`task_type = intake_create_or_update`

Приложение должно:

* создать карту, если её нет
* либо обновить существующую карту
* разложить данные по соответствующим секциям

---

## 7.3 Заметка после встречи

Пример:

```text
Наталья 3445
1. На МАК нашли демонстративных людей.
2. Д/З понаблюдать за тем, как «выпендриваешься» в жизни.

И далее сразу корректирую план на следующую встречу:

1. таблица самоощущений
2. Д/З – зачем ты выпендриваешься?
3. Рисунок семьи, рисунок школы на МАК.
4. Обсудить, что происходит в отношениях с отцом, с матерью, с учителями.
```

`task_type = session_update`

Приложение должно:

* добавить содержание встречи в секцию `Примечания по встречам`
* добавить план на следующую встречу в секцию `План следующей встречи`

---

## 7.4 Методическое обновление плана клиента

Это отдельный сценарий, когда на основе карты клиента и новых заметок нужно:

* сформировать мишени терапии
* обновить гипотезы
* предложить этапы работы
* построить или скорректировать план клиента

`task_type = client_plan_update`

# 8. Архитектура LLM

## 8.1 Один провайдер — Claude

На первый релиз используется один LLM-провайдер — Claude (Anthropic API).

Claude выполняет все задачи:

* первичную структуризацию intake-заметки
* структурирование заметки по встрече
* формирование и обновление плана клиента
* формирование мишеней терапии
* обновление гипотез
* обновление этапов работы
* очистку и форматирование текста
* возврат structured JSON

### Почему один провайдер

* Объём запросов для одного психолога — единицы в день. Разница в цене между моделями несущественна
* Один API-ключ, один формат промптов, одна обработка ошибок
* Claude справляется и с простыми, и со сложными задачами
* Меньше кода — меньше багов

### Возможное расширение

Если в будущем понадобится второй провайдер (OpenAI, Ollama), достаточно будет реализовать интерфейс `LLMProvider` и добавить роутер. Архитектура это позволяет, но реализовывать заранее не нужно.

---

## 8.2 Промпты

Все промпты хранятся в папке `prompts/` в формате `.md`.

Каждый промпт соответствует типу задачи:

* `intake_structuring.md` — для `intake_create_or_update`
* `session_note.md` — для `session_update`
* `client_plan.md` — для `client_plan_update`
* `section_update.md` — для точечного обновления секции
* `text_cleanup.md` — для очистки текста

---

# 9. Формат JSON от LLM

LLM должна возвращать строго structured JSON.

## 9.1 Для первичного создания/обновления карты

Пример:

```json
{
  "task_type": "intake_create_or_update",
  "client_key": "Анна 2332",
  "create_if_missing": true,
  "sections": {
    "Данные клиента": {
      "Пол и возраст": "женщина, 36 лет"
    },
    "Жалобы и запрос": {
      "Жалобы": "Амбивалентные чувства, не понимаю себя...",
      "Запрос": "Разобраться в себе"
    },
    "Первичная клиническая картина": {
      "Выявленный симптом": "ожидание помощи от окружающих",
      "Дополнения из жизни": "незавершённый развод..."
    }
  }
}
```

---

## 9.2 Для заметки после встречи

Пример:

```json
{
  "task_type": "session_update",
  "client_key": "Наталья 3445",
  "create_if_missing": false,
  "sections": {
    "Примечания по встречам": {
      "append_text": "На МАК нашли демонстративных людей..."
    },
    "План следующей встречи": {
      "append_list": [
        "таблица самоощущений",
        "Д/З – зачем ты выпендриваешься?",
        "Рисунок семьи, рисунок школы на МАК",
        "Обсудить отношения с отцом, матерью, учителями"
      ]
    }
  }
}
```

### Операции над полями

Каждое поле в `sections` может использовать одну из операций:

* **Строковое значение** (`"value": "..."`) — заменить содержимое поля
* **`append_text`** — добавить текст в конец существующего содержимого
* **`append_list`** — добавить пункты в список

---

## 9.3 Валидация JSON

Любой ответ LLM должен валидироваться через схему (pydantic).

Если JSON невалидный:

* не записывать в карту
* показать ошибку
* сохранить raw output в лог
* предложить повторить обработку

# 10. Pipeline работы приложения

## 10.1 Текстовый ввод

1. пользователь вставляет текст заметки
2. пользователь выбирает тип задачи (intake / session / plan)
3. приложение берёт первую строку
4. валидирует формат клиента
5. определяет, есть ли карта клиента
6. если карты нет — создаёт новую по шаблону
7. отправляет текст в LLM с нужным промптом
8. получает JSON
9. валидирует JSON
10. показывает preview изменений
11. делает backup
12. записывает изменения в карту
13. пишет лог

---

## 10.2 Голосовой ввод

1. пользователь загружает аудиофайл или записывает голос в приложении
2. аудио отправляется в speech-to-text
3. приложение показывает расшифрованный текст
4. пользователь подтверждает или исправляет текст
5. дальше идёт стандартный текстовый pipeline (с шага 2 из 10.1)

---

# 11. Голосовые заметки

Streamlit должен поддерживать два режима:

* загрузка файла через `st.file_uploader`
* запись через `st.audio_input`

Поддерживаемые форматы:

* `.wav`
* `.mp3`
* `.m4a`

Также можно опционально сохранять исходные файлы в:

* `data/audio_archive/`
* `data/inbox_notes/`

# 13. Backup

Перед любой записью создаётся backup текущей карты.

Пример:

```text
data/backups/anna_2332/anna_2332_2026-03-16_22-40-10.docx
```

Имена бэкапов используют латиницу (client_id), чтобы избежать проблем с кодировкой в файловой системе.

Нужен safe write:

1. сначала временный файл
2. потом замена оригинала только после успешного сохранения

* извлечение секций и полей по заголовкам (Heading 1 / Heading 2)

* отображение карты клиента в интерфейсе
* проверку, что шаблон читается корректно

На этом этапе пока не писать в карту автоматически.

## Этап 4. LLM, preview, backup и запись в карту

# 17. Ограничения первого релиза

На первом релизе не нужно:

* многопользовательский режим
* облачную базу
* внешний веб-доступ
* сложную векторную память
* автоматическую отправку данных куда-либо
* полную "умную" переработку всего документа целиком
* автоматическое определение типа задачи (пользователь выбирает вручную)

Приоритет первого релиза:

* локальность
* безопасность
* создание новой карты по шаблону
* поиск существующей карты
* аккуратное обновление нужных секций
* preview
* backup

# 18. Критерий готовности проекта

Проект можно считать рабочим, если психолог может:

1. открыть локальное приложение
2. вставить текст или загрузить/записать голос
3. выбрать тип задачи (первичная карта / заметка по встрече / план)
4. начать заметку с формата `Имя 1234`
5. получить найденную существующую карту или новую карту по шаблону
6. увидеть, как заметка разложилась по секциям
7. подтвердить preview
8. сохранить изменения в карту без повреждения `.docx`
